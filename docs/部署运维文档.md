# 部署运维文档

## 环境准备

推荐环境：

- Windows 10/11 或 Windows Server。
- Python 3.10 及以上。
- MySQL 8.x。
- 可选：MQTT Broker，例如 Mosquitto。

安装依赖：

```powershell
pip install -r requirements.txt
```

建议部署前确认：

- `python --version` 为 3.10 或以上。
- MySQL 服务已启动，且账号具备建表和读写权限。
- 生产环境已经准备独立的 `APP_SECRET_KEY` 和 `APP_CREDENTIAL_SECRET`。
- 如需 MQTT 接入，Broker 已启动并允许系统所在主机连接。
- 前端访问地址已加入 `FRONTEND_ORIGINS`。

## 数据库准备

默认使用 MySQL：

```text
DB_BACKEND=mysql
MYSQL_HOST=127.0.0.1
MYSQL_PORT=3306
MYSQL_DATABASE=gas_monitor
MYSQL_USER=root
MYSQL_PASSWORD=wcy20031001WCY
```

首次部署前创建数据库：

```sql
CREATE DATABASE gas_monitor CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

本地临时验证可切换 SQLite：

```powershell
$env:DB_BACKEND="sqlite"
.\start_system.bat
```

SQLite 适合本地演示和快速回归；正式部署建议使用 MySQL，便于多用户、长期历史数据和模型版本维护。

## 环境变量清单

| 类型 | 变量 | 说明 |

## 启动服务

系统服务：

```powershell
.\start_system.bat
```

前端服务：

```powershell
.\start_frontend.bat
```

虚拟设备、持续训练、漂移监控和 MQTT 网关在系统设置页通过开关启停。

生产环境可将上述命令交给进程管理工具、计划任务或 Windows 服务托管。托管时应显式设置工作目录为项目根目录，否则模型、数据库和静态资源路径可能解析错误。

## 运行检查

访问：

```text
前端：http://127.0.0.1:5173
后端：http://127.0.0.1:5000
```

重点检查：

- 独立前端 `http://127.0.0.1:5173` 是否正常显示登录页。
- 仪表盘是否能加载设备、告警和工单数据。
- `/api/dashboard` 是否返回 JSON。
- 虚拟设备启动后，历史数据是否持续增加。
- 单台设备累计达到 24 条读数后，历史数据是否出现异常得分、阈值、预测标签和模型版本。
- 告警列表是否能显示异常类型、异常得分、阈值、严重程度和模型版本。
- 设置页是否能显示训练、漂移监控和模型状态。

## 发布检查清单

1. 运行 `.\validate_project_health.bat`。
2. 使用目标数据库启动后端，确认数据库表可初始化。
3. 启动前端并访问 `http://127.0.0.1:5173`。
4. 登录后检查 `/api/auth/session`、`/api/dashboard`、`/api/settings`。
5. 注册一台测试设备并连续上报读数。
6. 验证告警、工单、历史数据、报表和重构曲线页面。
7. 检查 `APP_CREDENTIAL_SECRET` 是否为生产固定值。
8. 检查 `models/` 和 `models/registry/` 是否存在可用 active model；若不存在，确认系统能以规则逻辑降级运行。
9. 若开启 MQTT，发布注册和上传主题消息验证链路。
10. 备份部署时使用的环境变量、数据库连接信息和模型版本信息。

## 运行生成物

以下文件和目录由系统运行产生，不需要手动维护：

```text
data/raw/
data/processed/
data/experiment_results/
models/
models/registry/
instance/
```

如果本地测试数据混乱，可以在停止服务后清理上述生成物，再重新启动系统。清理前应确认不需要保留历史数据和模型版本。

## 备份与恢复

- 数据库应定期备份，重点保留设备、历史读数、告警、工单、人工反馈和模型元数据。
- `models/` 和 `models/registry/` 应随数据库一起备份，否则历史检测记录中的模型版本可能无法复现。
- `APP_CREDENTIAL_SECRET` 必须与数据库备份同时保管；丢失后，用户和工程师敏感字段无法解密。
- 恢复时先恢复环境变量和模型目录，再启动后端连接数据库。

## 运维建议

- 生产环境必须覆盖 `APP_SECRET_KEY` 和 `APP_CREDENTIAL_SECRET`。
- `APP_CREDENTIAL_SECRET` 用于字段级加密，部署后不要随意更换；更换前必须先完成密钥轮换迁移。
- MySQL 密码不要写入公开仓库，建议使用系统环境变量注入。
- 持续训练建议单独进程运行，不要和 Web 服务混在同一个命令窗口中。
- MQTT 网关需要先确认 Broker 可连接。
- 若模型误报明显增多，优先检查近期数据分布、阈值和人工反馈样本。
- 若检测结果长时间没有模型得分，检查设备是否已达到 24 点滑动窗口长度，以及 active model/scaler 是否存在。
- 若 Web 页面实时刷新异常，检查 WebSocket 连接和浏览器控制台错误。

## 日常巡检

| 检查项 | 建议频率 | 判断标准 |
| --- | --- | --- |
| 后端进程 | 每日 | `http://127.0.0.1:5000/api/dashboard` 可返回 |
| 前端页面 | 每日 | 登录页和主要菜单可打开 |
| 数据增长 | 每日 | 物理或模拟设备历史读数持续新增 |
| 告警数量 | 每日 | 无异常暴增或长时间为零 |
| 模型状态 | 每周 | active model、scaler 和注册表存在 |
| 漂移状态 | 每周 | 漂移得分无持续超阈值 |
| 数据库容量 | 每周 | 历史表和日志未占满磁盘 |
| 备份可用性 | 每月 | 数据库和模型备份可恢复 |

## 常见问题

### 依赖缺失

脚本提示缺少依赖时执行：

```powershell
pip install -r requirements.txt
```

### MySQL 连接失败

检查：

- MySQL 服务是否启动。
- 数据库 `gas_monitor` 是否存在。
- 用户名和密码是否正确。
- `MYSQL_HOST`、`MYSQL_PORT` 是否正确。

### 模型文件不存在

模型文件是生成物。可启动持续训练脚本生成候选模型，也可以先使用规则逻辑和已有数据运行系统。

### 虚拟设备没有数据

检查：

- Web 后端是否已经启动。
- 数据库配置是否和模拟器一致。
- `SIMULATION_DEVICE_COUNT` 是否为正数。

### 前端无法访问后端

检查：

- 后端是否在 `FLASK_HOST` 和 `FLASK_PORT` 指定地址启动。
- `FRONTEND_URL` 是否与实际前端地址一致。
- `FRONTEND_ORIGINS` 是否包含当前浏览器来源。
- 浏览器控制台是否出现跨域、网络或 WebSocket 错误。

### 登录后用户信息乱码或无法显示

检查：

- `PYTHONUTF8=1` 是否设置。
- `APP_CREDENTIAL_SECRET` 是否与创建用户时一致。
- 数据库字符集是否为 `utf8mb4`。

### 设备上报返回 pending 后页面没有结果

检查：

- `ASYNC_DETECTION_ENABLED` 是否开启。
- 异步检测线程数 `ASYNC_DETECTION_WORKERS` 是否合理。
- 单台设备是否达到 24 条连续读数。
- 后端日志中是否有模型文件、scaler 或数据库写入错误。

接口由 `src/app.py` 提供。页面路由用于浏览器访问，`/api/*` 路由用于前端和设备交互。

## 设备接入接口

### 设备注册

```http
POST /api/device/register
Content-Type: application/json
```

典型请求：

```json
{
  "meter_id": "GM9001",
  "name": "一号燃气表",
  "location": "太原市小店区",
  "protocol": "HTTP"
}
```

### 设备数据上报

```http
POST /api/device/upload
X-API-Key: <设备密钥>
Content-Type: application/json
```

典型请求：

```json
{
  "meter_id": "GM9001",
  "timestamp": "2026-05-04T08:30:00",
  "instant_flow": 0.36,
  "cumulative_usage": 128.52,
  "battery_voltage": 3.42,
  "signal_strength": 76.0,
  "valve_state": 1,
  "temperature": 22.5,
  "pressure": 2.1
}
```

系统处理流程：

1. 校验设备密钥。
2. 校验字段完整性和数据类型。
3. 写入历史监测表。
4. 按设备构造最近 24 点窗口。
5. 提取瞬时流量、累计用气量、电池电压、信号强度、阀门状态、环境温度和管网压力 7 类模型特征。
6. 调用 LSTM AutoEncoder 检测，计算重构误差和异常得分。
7. 结合低电压、弱信号、流量突变、长时间静止和压力异常等业务规则复核。
8. 写回异常得分、阈值、预测标签和模型版本。
9. 必要时生成告警和工单。

### 运营商 Webhook

```http
POST /api/carrier/webhook/<provider>
X-Carrier-Token: <可选平台令牌>
Content-Type: application/json
```

用于接收运营商平台转发的数据。若设置 `CARRIER_WEBHOOK_TOKEN`，请求必须携带一致的 `X-Carrier-Token`。

## 通用认证与权限

系统同时面向浏览器用户和物联网设备，因此认证方式分为两类。

浏览器用户接口使用 Session：

- `/api/auth/login` 登录后写入 Session。
- `/api/auth/session` 查询当前登录用户。
- `/api/auth/logout` 注销登录。
- 管理接口通常需要用户已登录。
- 用户、工程师、工单等接口会根据角色控制可见和可操作范围。

设备接口使用设备密钥：

- `/api/device/register` 返回或保存设备密钥。
- `/api/device/upload` 必须在请求头携带 `X-API-Key`。
- 后端根据 `meter_id` 和密钥匹配设备，匹配失败拒绝写入。

运营商 Webhook 使用可选平台令牌：

- 若配置 `CARRIER_WEBHOOK_TOKEN`，请求必须携带 `X-Carrier-Token`。
- 若未配置，接口仍可用于本地联调，但生产环境建议配置。

## 通用响应约定

接口优先返回 JSON。常见成功响应结构：

```json
{
  "ok": true,
  "data": {}
}
```

部分接口直接返回业务对象或数组，例如仪表盘、设备列表、告警列表和工单列表。前端统一通过 `request()` 解析 JSON，并在失败时显示错误消息。

常见错误状态：

| 状态码 | 场景 |
| --- | --- |
| `400` | 请求字段缺失、格式错误、状态流转不合法 |
| `401` | 未登录、设备密钥错误或运营商令牌错误 |
| `403` | 当前角色无权操作 |
| `404` | 设备、用户、工程师、告警或工单不存在 |
| `500` | 数据库、模型、后台服务或未知异常 |

错误响应通常包含 `error` 或 `message` 字段，前端会读取并展示。

## 关键接口字段说明

### 设备注册字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `meter_id` | string | 是 | 设备唯一编号 |
| `name` | string | 否 | 设备名称 |
| `location` | string | 否 | 安装位置或区域 |
| `protocol` | string | 否 | 接入协议，例如 HTTP、MQTT |
| `api_key` | string | 否 | 设备通信密钥；不传则由系统生成 |

### 设备上报字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `meter_id` | string | 是 | 设备编号 |
| `timestamp` | string | 否 | 采集时间，不传时使用服务端时间 |
| `instant_flow` | number | 是 | 瞬时流量 |
| `cumulative_usage` | number | 是 | 累计用气量 |
| `battery_voltage` | number | 是 | 电池电压 |
| `signal_strength` | number | 是 | 信号强度 |
| `valve_state` | number/bool | 是 | 阀门状态 |
| `temperature` | number | 是 | 环境温度 |
| `pressure` | number | 是 | 管网压力 |

系统用于模型检测的 7 个特征为：`instant_flow`、`cumulative_usage`、`battery_voltage`、`signal_strength`、`valve_state`、`temperature`、`pressure`。

## 设备上报响应示例

```json
{
  "ok": true,
  "reading_id": 1024,
  "meter_id": "GM9001",
  "detection_status": "pending",
  "message": "reading accepted"
}
```

`detection_status` 可能为：

- `pending`：读数已入库，异步检测尚未完成。
- `completed`：检测已完成，读数记录已有异常得分和预测标签。
- `skipped`：数据不足完整窗口或未启用模型检测，仅做基础规则判断。

## WebSocket 事件说明

WebSocket 用于减少页面轮询。后端在设备状态、读数、告警、工单、报表或设置状态变化时推送事件。前端收到事件后会根据事件类型刷新对应页面状态。

常见用途：

- 新读数入库后刷新仪表盘和历史曲线。
- 新告警生成后刷新告警列表和趋势图。
- 工单状态变化后刷新工单列表。
- 模型训练或漂移监控状态变化后刷新设置页。

## 接口联调建议

1. 先调用 `/api/device/register` 创建设备并保存密钥。
2. 连续调用 `/api/device/upload` 上传至少 24 条读数。
3. 查看 `/api/history` 是否出现异常得分、阈值和预测标签。
4. 查看 `/api/reconstruction` 是否返回原始序列、重构序列和误差。
5. 构造低电压、弱信号、压力异常或流量突变数据，查看 `/api/alerts`。
6. 从告警创建工单，调用接单和完成接口验证闭环。

## 典型联调命令

以下命令适合在后端启动后快速验证设备接入链路。PowerShell 中可根据实际返回替换设备密钥。

```powershell
$device = Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:5000/api/device/register" -ContentType "application/json" -Body '{"meter_id":"GM-TEST-001","name":"测试燃气表","location":"测试小区","protocol":"HTTP"}'
$apiKey = $device.api_key
```

```powershell
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:5000/api/device/upload" -Headers @{"X-API-Key"=$apiKey} -ContentType "application/json" -Body '{"meter_id":"GM-TEST-001","instant_flow":0.36,"cumulative_usage":128.52,"battery_voltage":3.42,"signal_strength":76,"valve_state":1,"temperature":22.5,"pressure":2.1}'
```

若设备注册响应结构发生调整，应以实际响应中的密钥字段为准。连续上传 24 条同一设备读数后，再检查 `/api/history` 和 `/api/reconstruction`。

## 接口变更维护规则

- 新增 `/api/*` 路由时，同步更新本文件的接口表、字段说明和页面映射。
- 修改请求字段时，同步更新前端 `request()` 调用和测试文档中的示例请求。
- 修改响应字段时，同步检查仪表盘、历史数据、告警、工单、报表和设置页是否仍能渲染。
- 修改认证方式时，同步更新安全与加密说明、部署运维文档和答辩系统讲解文档。
- 删除接口前先确认前端、脚本、MQTT、运营商 Webhook 和健康检查没有依赖。

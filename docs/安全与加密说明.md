# 安全与加密说明

## 加密目标

系统对管理员和工程师身份数据执行字段级加密，避免数据库中直接保存用户名、姓名、工号、电话、工程师片区、地址和头像等敏感信息。

## 实现方式

- 加密工具位于 `src/security.py`。
- 加密密钥由环境变量 `APP_CREDENTIAL_SECRET` 派生。
- 加密算法使用 Fernet 对称加密。
- 登录、查重和派单筛选使用 SHA-256 lookup 字段。
- 旧的短 VARCHAR 字段只保存不可读占位值，例如 `user-<hash>`、`emp-<hash>`、`[encrypted]`，避免短字段长度不足导致密文写入失败。

## 用户表字段

`users` 表保留业务兼容字段，同时新增：

- `username_encrypted`
- `username_lookup`
- `full_name_encrypted`
- `employee_no_encrypted`
- `employee_no_lookup`
- `phone_encrypted`
- `avatar_data_encrypted`

## 工程师表字段

`engineers` 表新增：

- `employee_no_encrypted`
- `employee_no_lookup`
- `avatar_data_encrypted`
- `name_encrypted`
- `phone_encrypted`
- `address_encrypted`
- `region_encrypted`
- `region_lookup`

## 运行迁移

系统启动时会在数据库初始化阶段补齐缺失字段，并将历史明文身份数据迁移到加密字段。迁移后业务接口仍返回解密后的展示值，前端无需感知数据库存储方式。

## 运维要求

- 生产环境必须覆盖默认 `APP_CREDENTIAL_SECRET`。
- 生产环境也必须覆盖默认 `APP_SECRET_KEY`、数据库密码和 MQTT/Webhook 等外部接入凭据。
- 密钥必须长期稳定保存，不能随意更换。
- 如果需要更换密钥，应增加密钥轮换脚本，按“旧密钥解密，新密钥加密”的方式迁移。
- 数据库备份应与应用密钥分开保存。
- 答辩演示可使用本地默认账号和本地数据库；正式部署不得沿用演示密钥和演示密码。

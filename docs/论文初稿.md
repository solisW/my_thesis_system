# 基于深度学习的物联网智能燃气表数据异常检测与调度系统设计与实现（初稿）

## 摘要

随着城市燃气行业向数字化、智能化方向发展，物联网智能燃气表在远程抄表、异常预警、设备维护等业务中发挥着重要作用。传统的人工巡检和静态阈值判断方法难以及时、准确地识别多设备并发场景下的异常状态。为此，本文设计并实现了一个基于深度学习的物联网智能燃气表异常检测与调度系统。

系统以多源设备数据为输入，采用 LSTM AutoEncoder 对智能燃气表运行时序进行建模，通过重构误差实现异常检测；同时结合实时监控后台、设备管理、地图告警、工程师管理和工单调度，形成“感知-分析-告警-处置”的闭环业务流程。系统支持 50 台模拟设备持续上报，也保留了与真实物理设备对接的标准接口，能够满足课程设计、毕业设计与后续工程扩展的需求。

关键词：物联网；智能燃气表；异常检测；深度学习；LSTM AutoEncoder；工单调度

## 1. 绪论

### 1.1 研究背景

智能燃气表是智慧燃气建设中的关键终端设备，能够对燃气流量、累计用气量、电池状态、通信信号和阀门状态等信息进行持续采集与上报。随着部署规模不断扩大，海量设备的在线状态、通信质量和异常行为识别变得更加复杂。若仍依赖人工经验或简单阈值规则，不仅检测精度有限，而且难以支撑大规模实时监测。

因此，构建一个面向智能燃气表场景的异常检测与调度平台具有重要意义。一方面，它可以通过深度学习方法自动识别异常模式；另一方面，它可以将检测结果与工程运维流程联动，提高故障处置效率。

### 1.2 研究意义

本课题的研究意义主要体现在以下几个方面：

1. 提高设备异常识别的准确性与实时性；
2. 降低人工巡检成本，提升运维效率；
3. 为燃气企业设备管理和风险预警提供可视化支撑；
4. 为物联网设备监测与智能调度系统建设提供工程实践参考。

## 2. 系统需求分析

### 2.1 功能需求

根据实际业务场景，系统需满足如下功能需求：

1. 用户登录与后台访问控制；

3. 主页显示在线设备数、离线设备数和异常设备数；

- 实时性：设备数据变化后应尽快反馈至前端；

## 3. 系统总体设计

### 3.1 系统架构

系统总体架构分为四层：

1. 设备接入层  
   包括模拟燃气表与真实物理设备。模拟设备用于持续生成测试数据，真实设备通过标准上传接口接入。

2. 数据处理层  
   后端接收设备数据后完成持久化、异常检测、状态更新和告警生成。

3. 业务管理层  
   包括设备管理、工程师管理、工单调度、告警中心和统计报表等模块。

4. 前端展示层  
   通过后台菜单进入各功能页面，并使用 WebSocket 实现实时刷新和状态推送。

### 3.2 功能模块设计

系统主要包括以下模块：

- 登录认证模块
- 首页总览模块
- 地图告警模块
- 设备管理模块
- 工程师管理模块
- 工单调度模块
- 历史数据模块
- 统计报表模块
- 模拟设备模块
- 真实设备接入模块

## 4. 关键技术设计

### 4.1 深度学习异常检测

系统采用 LSTM AutoEncoder 模型对智能燃气表多维时序特征进行建模。模型先对正常数据进行训练，学习燃气设备在稳定状态下的时间序列模式；在推理阶段，若输入数据无法被模型较好重构，则说明其偏离正常模式，异常分数升高。

异常分数采用均方误差定义：

```text
Score = mean((X - X_hat)^2)
```

其中，`X` 为原始输入窗口，`X_hat` 为模型重构结果。

### 4.2 WebSocket 实时更新

为提升前端刷新流畅度，系统不再使用固定轮询方式，而是采用 WebSocket 建立实时推送通道。后端在每轮模拟设备批量写入完成后统一广播最新状态，从而降低前端频繁重绘和后端无效推送带来的性能开销。

### 4.3 批量模拟设备机制

系统默认创建 50 台模拟设备，每台设备按照预设的燃气使用规律生成流量、电压、信号、温度和压力等数据，并按固定间隔持续上报。相比单设备测试，多设备并发更接近实际部署场景，有利于验证系统稳定性和实时性。

### 4.4 工单联动机制

当设备被识别为异常后，管理人员可在后台为异常设备创建紧急工单，并分配给具体工程师处理。通过工程师管理模块维护人员信息，实现告警与运维流程的联动。

## 5. 数据库设计

系统数据库包含以下核心数据表：

1. 用户表 `users`  
   存储管理员和系统用户信息。

2. 设备表 `devices`  
   存储设备编号、位置、协议、经纬度、运行模式等基础信息。

3. 采集数据表 `meter_readings`  
   存储设备持续上报的多维运行数据。

4. 异常事件表 `anomaly_events`  
   存储异常类型、异常分数、告警描述和严重等级。

5. 工程师表 `engineers`  
   存储工程师姓名、电话、专长、状态和负责片区。

6. 工单表 `work_orders`  
   存储工单标题、描述、优先级、状态以及设备与工程师的关联关系。

## 6. 系统实现

### 6.1 首页总览

系统主页显示在线设备数量、离线设备数量、异常设备数量和设备总数，同时通过地图面板显示异常设备所在位置，并通过颜色区分在线、离线和异常状态。

### 6.2 设备管理

设备管理页面支持对单个设备进行信息维护，包括新增、编辑、删除，并提供连通性测试功能。对于真实设备，可通过 IP 与端口配置进行测试；对于持续上报中的设备，可通过最近心跳判断连通性。

### 6.3 工程师管理与工单调度

系统设置了工程师管理页面，用于维护工程师信息；在工单调度页面中，管理人员可以创建工单、设置优先级、选择关联设备，并将工单分配给具体工程师。

### 6.4 模拟设备与真实设备共存

系统保留了真实物理设备上传接口，接口采用 `POST /api/device/upload`，并通过 `X-API-Key` 完成设备认证。模拟设备与真实设备共用同一套后端处理链路，因此系统具备从演示环境向真实环境扩展的能力。

## 7. 系统测试与分析

### 7.1 功能测试

通过系统功能测试可验证以下结果：

- 登录页面可正常访问；
- 菜单栏可跳转至各功能页面；
- 50 台模拟设备能够持续上报；
- 设备状态、异常数量和地图位置可实时更新；
- 设备管理、工程师管理和工单调度功能可正常使用；
- 真实设备接口可接收标准 JSON 数据。

### 7.2 性能与流畅性分析

系统在设计中采用了“批量上报 + WebSocket 推送 + 前端节流渲染”策略。相比对每条设备数据单独广播的方式，该方案显著减少了前端重复渲染次数，在 50 台设备持续上报场景下能够保持较好的流畅性。

## 8. 总结与展望

本文完成了一个面向智能燃气表场景的异常检测与调度系统设计与实现。系统不仅集成了深度学习异常检测模型，还将设备监测、地图告警、工程师管理和工单调度纳入统一平台，实现了从数据采集到异常处置的完整闭环。

5. 实现移动端巡检与工单反馈功能。

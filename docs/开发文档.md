# 开发文档

## 开发原则

- 代码和文档以论文初稿《基于深度学习的物联网智能燃气表数据异常检测系统的设计与实现》为业务基准，新增功能应能归入数据接入与预处理、异常检测引擎、告警工单、监控看板或持续训练流水线。
- 系统运行逻辑和论文材料分离，`docs/` 中不再保留论文图片、PPT 工作区或渲染中间产物。
- 运行生成物不入库，包括数据库、模型文件、CSV 数据、缓存和 `__pycache__`。
- 新增功能优先放在已有业务模块内，避免把逻辑全部堆到路由函数中。
- 路由层只做 HTTP/Session/WebSocket 入口处理；后台服务启停、实时广播等跨业务编排放入 `src/application/`。
- 面向设备、身份、监控和工单的调用优先通过 `src/domain/` 门面进入，减少上层直接依赖 `services.py` 内部函数。
- 设备接入统一走 `device_integration.py`，避免 HTTP、MQTT、运营商数据各自实现一套检测逻辑。
- 模型版本统一走 `model_registry.py`，避免直接覆盖线上模型文件。

## 本地开发流程

1. 安装依赖：

```powershell
pip install -r requirements.txt
```

2. 使用 SQLite 快速启动：

```powershell
$env:DB_BACKEND="sqlite"
.\start_system.bat
```

3. 登录系统后进入“系统设置”，按需开启虚拟设备、持续训练、漂移监控和 MQTT 网关。

4. 运行健康检查：

```powershell
.\validate_project_health.bat
```

5. 前端静态服务单独启动：

```powershell
.\start_frontend.bat
```

开发时默认后端地址为 `http://127.0.0.1:5000`，前端地址为 `http://127.0.0.1:5173`。若修改端口，应同步检查 `FRONTEND_URL`、`FRONTEND_ORIGINS`、`FLASK_HOST` 和 `FLASK_PORT`。

## 代码目录职责

| 路径 | 职责 | 维护要点 |
| --- | --- | --- |
| `src/app.py` | Flask 应用创建、页面跳转、API 路由、认证权限、WebSocket 路由 | 路由层只做认证、参数接收和响应封装，复杂业务下沉到应用层或领域层 |
| `src/application/runtime.py` | 模拟器、持续训练、MQTT 网关、漂移监控等后台服务运行态 | 新增长生命周期服务时在这里统一创建、复用和启停 |
| `src/application/realtime.py` | WebSocket 广播编排 | 路由和服务状态变化后复用这里的广播函数，避免重复拼装刷新事件 |
| `src/domain/` | 设备、身份、监控、工单、初始化等业务域门面 | 上层调用优先依赖门面，底层实现可逐步从 `services.py` 拆细 |
| `src/services.py` | 设备、用户、工程师、告警、工单、报表、设置等核心业务服务 | 保持业务规则单一来源，避免前端和接口各自实现规则 |
| `src/database.py` | SQLAlchemy 模型和数据库初始化 | 修改表结构后同步更新接口、测试和部署说明 |
| `src/device_integration.py` | 设备注册、上报和统一接入流程 | HTTP、MQTT、运营商和模拟器都应复用该入口 |
| `src/detection_service.py` | 在线异常检测和重构曲线计算 | 保持 `WINDOW_SIZE`、`FEATURE_COLUMNS`、模型文件和阈值逻辑一致 |
| `src/training_module.py` | 训练、评估和持续训练循环 | 候选模型写入注册表，避免直接覆盖 active model |
| `src/model_registry.py` | 模型版本注册、查询和切换 | 记录模型指标、阈值、文件路径和激活状态 |
| `src/simulation_module.py` | 虚拟设备数据生成 | 用于答辩演示和本地回归，不应替代真实设备接入测试 |
| `src/mqtt_gateway.py` | MQTT 注册和上报主题消费 | 只负责消息解析和转发，不重复实现检测逻辑 |
| `src/carrier_gateway.py` | 运营商 Webhook 数据适配 | 将运营商字段映射为系统标准字段 |
| `src/drift_monitor.py` | 数据漂移检测和状态维护 | 漂移结果用于提醒复训，不直接替代异常检测结果 |
| `frontend/src/api.js` | HTTP JSON 请求封装和统一错误处理 | 新增接口调用时优先复用该文件 |
| `frontend/src/config.js` | 页面元数据、运行配置和地图默认参数 | 新增页面或调整前端运行参数时同步更新 |
| `frontend/src/formatters.js` | 默认状态、表单初值和展示格式化函数 | 新增页面状态或显示格式时同步更新 |
| `frontend/src/map-utils.js` | 地图坐标解析和点位工具函数 | 调整地图点位时同步更新 |
| `frontend/src/main.js` | 前端页面状态、图表、地图和 WebSocket 编排 | 新增页面时同步更新页面与接口映射 |
| `frontend/src/styles.css` | 前端视觉样式 | 保持管理系统风格，避免影响移动端和表格可读性 |

## 数据库开发

数据库模型位于 `src/database.py`。主要实体包括：

- 用户。
- 设备。
- 工程师。
- 历史监测数据。
- 异常告警。
- 工单。
- 训练数据。
- 模型元数据。

开发时应保证：

- 设备读数能追溯到设备。
- 异常告警能追溯到读数和模型版本。
- 工单能追溯到告警和工程师。
- 人工反馈能回流到训练数据。

新增字段或实体时，应同步完成以下事项：

1. 在 `src/database.py` 中定义模型字段和关系。
2. 在 `services.py` 中补充序列化、创建、更新和查询逻辑。
3. 在 `docs/接口文档.md` 中补充请求字段、响应字段和错误场景。
4. 在 `docs/测试文档.md` 中补充对应的冒烟或回归检查。
5. 对生产库变更，先备份数据库，再执行迁移或初始化脚本。

### 身份数据加密

管理员和工程师身份数据采用“字段级加密 + 查询 lookup”的方式存储：

- `src/security.py` 使用 `APP_CREDENTIAL_SECRET` 派生 Fernet 密钥。
- 用户名、姓名、工号、电话、头像、工程师片区和地址写入 `*_encrypted` 字段。
- 登录、查重和派单筛选使用 SHA-256 lookup 字段，例如 `username_lookup`、`employee_no_lookup`、`region_lookup`。
- 旧短字段只保留不可读占位值，避免受 VARCHAR 长度限制，也避免数据库中出现明文敏感信息。
- 业务层统一通过 `services.py` 中的解密辅助函数读取展示值，不应在路由或前端直接依赖数据库旧明文字段。

生产环境必须固定并妥善保管 `APP_CREDENTIAL_SECRET`。更换该密钥会导致历史密文字段无法解密，除非先完成密钥轮换迁移。

## 异常检测开发

在线检测入口在 `src/detection_service.py`。核心输入是某台设备最近的连续读数，检测流程为：

1. 读取最近 `WINDOW_SIZE=24` 条数据。
2. 按 `FEATURE_COLUMNS` 取 7 个特征。
3. 使用模型对应的 StandardScaler 标准化。
4. 调用 LSTM AutoEncoder 重构。
5. 计算 MSE 重构误差。
6. 与模型阈值比较得到预测标签。
7. 结合低电压、弱信号、流量突变、压力异常等业务规则进行二次研判。
8. 返回异常得分、阈值、标签、异常类型、严重程度和模型版本。

开发注意事项：

- `FEATURE_COLUMNS` 必须与训练、推理、重构曲线和实验导出保持一致。
- 数据不足 24 条时不应强行推理，应返回跳过或等待状态。
- 模型文件缺失时系统应仍能启动，并允许规则检测和数据接入继续运行。
- 业务规则触发的异常需要写入可读的异常类型，便于告警页、工单页和答辩演示解释。
- 异步检测开启后，设备上报接口可能先返回 `pending`，前端通过历史数据刷新或 WebSocket 获取后续结果。

## 训练开发

训练相关模块：

- `training_data_generator.py`：生成训练窗口。
- `training_data_cleaner.py`：清洗训练数据。
- `training_repository.py`：读取和保存训练样本。
- `training_module.py`：训练模型、评估指标、持续训练循环。
- `model_registry.py`：注册候选模型和切换 active model。

训练生成物应写入 `models/` 或数据库，不应写入系统说明文档目录。

模型版本维护原则：

- 新模型先作为候选版本注册，记录训练时间、窗口长度、阈值、评估指标和文件路径。
- 只有指标满足切换条件或人工确认后，才切换为 active model。
- 切换模型后需要保留旧版本，便于误报增多时回退和答辩时解释版本来源。
- 若调整特征列、窗口长度或标准化方式，必须重新训练 scaler 和模型，不能复用旧 scaler。

## 接入开发

新增设备来源时，应尽量转换为统一的数据结构，再调用已有设备注册和上报流程。

现有来源：

- HTTP：`/api/device/register`、`/api/device/upload`。
- MQTT：`mqtt_gateway.py`。
- 运营商平台：`carrier_gateway.py`。
- 虚拟设备：`simulation_module.py`。

标准接入字段为：

| 字段 | 含义 |
| --- | --- |
| `meter_id` | 设备编号 |
| `timestamp` | 采集时间 |
| `instant_flow` | 瞬时流量 |
| `cumulative_usage` | 累计用气量 |
| `battery_voltage` | 电池电压 |
| `signal_strength` | 信号强度 |
| `valve_state` | 阀门状态 |
| `temperature` | 环境温度 |
| `pressure` | 管网压力 |

接入新协议时，只做字段映射、鉴权和异常保护，然后交给统一上报流程。这样可以保证历史数据、异常检测、告警、WebSocket 和报表都自动复用。

## 分层开发

新增后端能力时按以下顺序落位：

1. HTTP 请求、Session 权限、状态码和 JSON 响应放在 `src/app.py`。
2. 需要组合多个业务域、控制后台线程或推送实时事件的逻辑放在 `src/application/`。
3. 设备、身份、监控、工单等业务调用通过 `src/domain/` 暴露给上层。
4. 数据库模型、模型文件、MQTT、运营商 Webhook、训练数据和外部协议适配放在基础设施模块。

不建议在路由函数中直接管理长生命周期对象，也不建议从前端接口绕过统一设备接入、模型注册或工单闭环。

## 前端开发

- 前端为独立工程，不再使用 Flask 模板。
- 入口在 `frontend/index.html`。
- 前端运行配置和页面元数据在 `frontend/src/config.js`。
- HTTP JSON 请求封装在 `frontend/src/api.js`。
- 默认状态、表单初值和展示格式化函数在 `frontend/src/formatters.js`。
- 地图坐标解析和点位工具函数在 `frontend/src/map-utils.js`。
- 页面状态、WebSocket、ECharts 和 Leaflet 渲染编排在 `frontend/src/main.js`。
- 现代化视觉样式在 `frontend/src/styles.css`。
- 新页面应保持前后端分离，只通过 `/api/*` 和 `/ws` 与后端通信。
- 实时数据优先走 WebSocket，必要时使用定时刷新兜底。

前端新增功能流程：

1. 在 `api.js` 中增加或复用 API 调用方法。
2. 在 `formatters.js` 或 `main.js` 中增加默认状态、页面数据、加载状态和错误状态。
3. 在页面渲染区域增加视图和操作按钮。
4. 需要实时刷新的页面，订阅或复用现有 WebSocket 刷新逻辑。
5. 在 `styles.css` 中补充响应式样式。
6. 更新 `docs/接口文档.md` 的页面与接口映射。

## 提交前自查

- `.\validate_project_health.bat` 通过。
- 后端 `http://127.0.0.1:5000/api/dashboard` 能返回 JSON。
- 前端 `http://127.0.0.1:5173` 能打开登录页和主要页面。
- 设备注册、连续上报 24 条读数、告警生成、工单完成能跑通。
- 新增配置项已经写入 `docs/配置说明.md`。
- 新增或修改接口已经写入 `docs/接口文档.md`。
- 运行生成物没有放入 `docs/`。

## 文档维护

系统文档只记录可运行系统、配置、接口和运维方式。文档表述应与论文初稿的模块划分保持一致：数据接入与预处理、深度学习异常检测、告警与工单、实时监控看板、持续训练。若需要生成论文图片、PPT 或实验图，应放到 `data/` 或临时目录，使用后清理，不再纳入仓库文档结构。

答辩材料例外保留在 `docs/`：答辩稿、答辩 PPT、`答辩-系统总结与代码讲解.md` 和 `答辩-论文初稿总结与讲解.md`。这些文档应与当前源码、接口和启动方式同步更新。

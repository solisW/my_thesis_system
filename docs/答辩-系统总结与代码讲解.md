# 答辩用系统总结与代码讲解

本文档用于毕业答辩现场讲解系统工程实现、代码结构、核心调用链、关键文件职责和可能被追问的技术问题。它按照当前项目最终答辩版本重新构造，目标是让你可以快速回答“系统怎么跑、代码怎么写、为什么这么设计、出了问题怎么解释”。

## 1. 系统总体定位

本系统是一个面向物联网智能燃气表的异常检测与运维管理平台。它不是单纯的算法脚本，也不是普通后台管理系统，而是把设备接入、数据存储、深度学习检测、业务规则、告警、工单、前端可视化和持续训练组合成完整闭环。

一句话讲系统：

> 智能燃气表通过 HTTP、MQTT 或运营商 Webhook 上报多维时序数据，后端校验并入库后，按设备构造 24 点、7 特征窗口，调用 LSTM AutoEncoder 计算重构误差和异常得分，再结合业务规则生成可解释告警，并通过工单模块完成派单、接单、回单和人工反馈，前端通过 API 和 WebSocket 实时展示运行状态。

系统核心闭环：

```text
设备数据
  -> 接入校验
  -> 数据库存储
  -> 异常检测
  -> 告警生成
  -> 工单处理
  -> 人工反馈
  -> 持续训练
  -> 模型版本更新
```

## 4. 后端文件逐个讲解

### 4.1 `src/app.py`

职责：

- 创建 Flask 应用。
- 配置数据库、Session、CORS 和 WebSocket。
- 初始化数据库表、默认账号、模型元数据。
- 注册所有 HTTP API。
- 启动后台服务。
- 提供 `/ws` WebSocket 实时推送入口。

核心函数：

| 函数 | 功能 |
| --- | --- |
| `create_app()` | 应用工厂，创建 Flask 实例并初始化全部组件 |
| `initialize_database_with_retry()` | 数据库建表、迁移、默认数据和模型元数据同步，支持 MySQL 并发 DDL 重试 |
| `is_transient_mysql_startup_error()` | 判断 MySQL 启动阶段是否是可重试错误 |
| `register_routes(app)` | 注册所有页面跳转、认证接口、业务接口和 WebSocket |

核心代码逻辑：

```text
create_app()

  -> 设置 SECRET_KEY、SQLALCHEMY_DATABASE_URI、Session Cookie
  -> ensure_directories()

  -> db.init_app(app)
  -> sock.init_app(app)

  -> register_cors(app)
  -> register_routes(app)
```

答辩解释：

> `app.py` 是后端总入口。系统启动时不仅创建 Web 服务，还会完成数据库初始化、默认账号准备、模型元数据同步和后台服务启动。这样答辩演示时只需要运行统一启动脚本，系统就能进入可用状态。

可能追问：

| 问题 | 回答 |
| --- | --- |
| 为什么路由都在 `app.py`？ | 这是 Flask 入口层，负责 HTTP 接收和响应；具体业务通过 domain 和 services 调用。 |

### 4.6 `src/detection_service.py`

职责：

- 加载 LSTM AutoEncoder、scaler 和模型元数据。
- 在模型文件不存在时触发训练流水线。
- 对设备最近读数进行异常评分。
- 在数据不足 24 条时使用启动阶段规则评分。
- 返回重构曲线数据给前端。

核心类：

```text
DetectionResult
DetectionService
```

`DetectionResult` 字段：

| 字段 | 含义 |
| --- | --- |
| `anomaly_score` | 异常得分，即重构误差或启动阶段规则分 |
| `threshold` | 阈值 |
| `predicted_label` | 0 正常，1 异常 |
| `anomaly_type` | 异常类型 |
| `description` | 异常描述 |
| `severity` | 严重程度 |
| `model_version` | 模型版本 |

核心方法：

| 方法 | 功能 |
| --- | --- |
| `ensure_ready()` | 确保模型、scaler、meta 已存在并加载 |
| `score_recent_readings(readings)` | 对最近读数评分 |
| `reconstruction_trace(readings)` | 返回原始序列、重构序列和误差序列 |
| `_frame_from_readings(readings)` | 将数据库读数转为 DataFrame |
| `_bootstrap_score(latest, readings)` | 数据不足窗口时的规则评分 |
| `_has_operational_anomaly(latest, readings)` | 判断是否存在业务异常 |
| `_infer_anomaly_type(...)` | 根据业务规则输出异常类型和描述 |

完整检测逻辑：

```text
score_recent_readings()
  -> ensure_ready()
  -> if len(readings) < WINDOW_SIZE:
       使用 _bootstrap_score()
       使用 _infer_anomaly_type()
     else:
       _frame_from_readings()
       clean_dataset()
       transform_features()
       构造 tensor: [1, 24, 7]
       model(tensor)
       MSE = mean((reconstructed - tensor)^2)
       threshold = meta["threshold"]
       predicted_label = anomaly_score > threshold and _has_operational_anomaly()
       _infer_anomaly_type()
```

答辩解释：

> 检测服务把算法封装成统一接口。上层业务不需要关心 PyTorch 张量、scaler 和模型文件，只需要传入某台设备最近读数，就能得到异常得分、阈值、标签、异常类型和严重程度。

可能追问：

| 问题 | 回答 |
| --- | --- |
| 模型文件缺失怎么办？ | `ensure_ready()` 会触发训练流水线生成模型，保证系统可以恢复。 |
| 为什么数据不足时不直接模型推理？ | LSTM 输入需要完整窗口，数据不足时强行推理没有意义，所以用启动阶段规则评分。 |
| 重构曲线怎么来的？ | `reconstruction_trace()` 用模型重构最近窗口，再 inverse transform 回原始尺度供前端绘图。 |

### 4.7 `src/model.py`

职责：

- 定义 LSTM AutoEncoder 神经网络结构。

核心代码结构：

```text
class LSTMAutoEncoder(nn.Module):
    encoder = nn.LSTM(input_size, hidden_size, num_layers)
    decoder = nn.LSTM(hidden_size, hidden_size, num_layers)
    output_layer = nn.Linear(hidden_size, input_size)
```

`forward()` 逻辑：

```text
输入 inputs: [batch, time, feature]
  -> encoder(inputs)
  -> 取最后一层 hidden[-1]
  -> unsqueeze 成 [batch, 1, hidden]
  -> repeat 成 [batch, time, hidden]
  -> decoder(repeated_context)
  -> output_layer(decoded)
  -> 输出 [batch, time, feature]
```

答辩解释：

> 编码器把整个时间窗口压缩成隐藏表示，解码器根据这个隐藏表示重构整个窗口。重构结果和原始输入越接近，说明数据越符合正常模式；重构误差越大，说明越可能异常。

## 8. 核心调用链

### 8.1 用户登录调用链

```text
frontend login form
  -> ApiClient.request("/api/auth/login")
  -> app.py api_auth_login()
  -> User 表按 username_lookup 查询用户
  -> cipher.verify()
  -> open_user_session()
  -> api_auth_session()
  -> frontend 保存 user
  -> loadPage()
```

知识点：

- Session。
- 密码验证。
- 用户名 lookup。
- 角色权限。

可能追问：

| 问题 | 回答 |
| --- | --- |
| 为什么 username 要 lookup？ | 用户名加密后不能直接查询，使用哈希 lookup 做等值查询。 |
| 登录态如何保持？ | Flask Session 写入浏览器 Cookie，前端请求携带 credentials。 |

### 8.2 设备注册调用链

```text
POST /api/device/register
  -> app.py api_device_register()
  -> domain.devices.register_device_endpoint()
  -> device_integration.register_device_endpoint()
  -> Device 表查询 meter_id
  -> 不存在则生成 api_key 并创建 Device
  -> 存在则更新设备信息
  -> 返回 meter_id, api_key, protocol
```

知识点：

- 设备唯一编号。
- API Key。
- 幂等注册。
- 协议扩展。

### 8.3 设备上报和异常检测调用链

```text
POST /api/device/upload
  -> app.py api_device_upload()
  -> accept_device_reading(api_key, payload)
  -> Device 表校验 api_key
  -> _normalize_payload()
  -> services.ingest_reading()
  -> MeterReading 入库
  -> Device 更新 last_seen_at/status
  -> _submit_detection_task()
  -> _run_detection_for_reading_once()
  -> detector.score_recent_readings()
  -> 写回 anomaly_score/threshold/predicted_label/model_version
  -> 异常时创建 AnomalyEvent
  -> _auto_create_work_order_for_event()
  -> emit_realtime_updates()
  -> WebSocket 推送前端
```

这条链路是答辩最重要的代码链路。

答辩讲法：

> 系统把设备上报、模型推理、告警和工单串成一条完整链路。设备上报不是只写数据库，而是会触发检测，检测结果写回读数表，异常时生成告警并进入工单流程，最后推送给前端。

### 8.4 运营商 Webhook 调用链

```text
POST /api/carrier/webhook/<provider>
  -> app.py api_carrier_webhook()
  -> carrier_gateway.ingest_carrier_payload()
  -> token 校验
  -> _flatten_payload()
  -> _pick() 字段别名映射
  -> register_device_endpoint()
  -> accept_device_reading()
  -> 后续复用设备上报检测链路
```

答辩讲法：

> 运营商 Webhook 只负责适配第三方字段，最终仍进入统一设备上报流程。这体现了系统可扩展性。

### 8.5 MQTT 调用链

```text
系统设置页启动 MQTT
  -> /api/mqtt
  -> RuntimeServices.mqtt_service()
  -> MqttGateway.start()
  -> 订阅 register/upload 主题
  -> 收到消息后解析 JSON
  -> register_device_endpoint() 或 accept_device_reading()
  -> 后续复用检测链路
```

### 8.6 重构曲线调用链

```text
frontend history/reconstruction page
  -> GET /api/reconstruction
  -> services.reconstruction_snapshot()
  -> detector.reconstruction_trace()
  -> 查询最近 24 条读数
  -> clean_dataset()
  -> transform_features()
  -> model(tensor)
  -> scaler.inverse_transform()
  -> 返回 original/reconstructed/error_series
  -> ECharts 绘图
```

知识点：

- 重构曲线用于解释模型结果。
- 原始值和重构值要逆标准化回原始尺度。
- 误差曲线显示每个时间点的偏离程度。

### 8.7 工单流转调用链

```text
告警生成
  -> _auto_create_work_order_for_event()
  -> create_work_order()
  -> dispatch_candidate_engineers()
  -> WorkOrder 入库
  -> WorkOrderRecord 记录创建/派发
  -> 工程师登录
  -> engineer_accept_work_order()
  -> 状态 in_progress
  -> engineer_complete_work_order()
  -> 状态 completed
  -> AnomalyEvent handled_at / confirmed_label 更新
  -> WebSocket 推送
```

答辩讲法：

> 工单模块体现“异常可处置”。它不仅保存最终状态，还通过 `WorkOrderRecord` 记录每次流转，便于追踪。

### 8.8 持续训练调用链

```text
系统设置页启动训练
  -> /api/training
  -> RuntimeServices.trainer_service()
  -> ContinuousTrainingService.start()
  -> run_training_pipeline()
  -> generate_training_data()
  -> clean_training_data()
  -> build_windows()
  -> train LSTMAutoEncoder
  -> compute threshold
  -> evaluate candidate
  -> stage_model_bundle()
  -> should_activate()
  -> activate_model_bundle()
  -> ModelMetadata 更新
```

答辩讲法：

> 持续训练不是直接覆盖当前模型，而是训练候选模型，计算指标，经过模型注册和激活判断后再切换 active model，保证模型更新可控。

## 10. 系统中的关键知识点

- `request.get_json()` 获取 JSON 请求体。
- `jsonify()` 返回 JSON。
- 装饰器用于登录和角色权限。

- `db.relationship` 定义关系。
- `db.session.add()` 添加对象。
- `db.session.commit()` 提交事务。
- `db.select()` 和 Query 查询数据。

### 10.3 WebSocket

- HTTP 适合请求响应。
- WebSocket 适合实时推送。
- 系统通过 `hub.broadcast(event, payload)` 推送事件。
- 前端根据事件名更新页面状态。

### 10.4 PyTorch

- `nn.Module` 定义模型。
- `nn.LSTM` 处理序列。
- `torch.tensor()` 构造输入张量。
- `torch.no_grad()` 用于推理，避免计算梯度。

- 训练时拟合 scaler。

## 11. 高频代码答辩问题

### 11.1 为什么要有 `services.py`？

因为路由层只适合处理请求和响应，真正的业务流程包括设备状态、告警、工单、模型检测、实时推送等。如果全部写在 `app.py`，代码会非常难维护。`services.py` 把业务规则集中起来，使 HTTP、MQTT、Webhook、模拟器都能复用同一套逻辑。

### 11.2 为什么有 `domain/` 目录？

`domain/` 是业务门面层，让 `app.py` 引入设备、身份、监控、工单等业务入口时更清晰。底层目前复用成熟的 `services.py`，后续如果继续拆分服务，也可以从 `domain/` 逐步演进。

### 11.3 为什么检测要异步？

设备上报接口需要快速返回，模型推理和告警生成可能耗时。异步检测可以先保证数据入库，再由后台任务计算异常结果，提升接口响应速度。结果通过 reading_id 写回，不会丢失归属。

### 11.4 数据不足 24 条怎么办？

完整 LSTM 推理需要 24 点窗口。数据不足时系统使用启动阶段规则评分，例如低电压、弱信号、压力异常和流量突变。等数据达到窗口长度后再进入完整模型推理。

### 11.5 为什么要 `ModelMetadata`？

模型检测结果要可追溯。`ModelMetadata` 记录模型 ID、路径、scaler、阈值、特征、窗口长度、指标和激活状态。如果后续出现误报或需要回滚，可以知道当时用的是哪个模型。

### 11.6 为什么要人工反馈？

模型无法一次性完美。工程师处理后可以确认异常、误报或忽略，这些反馈可以沉淀为标注样本，用于后续评估、规则优化和持续训练。

### 11.7 为什么前端用 WebSocket？

设备、告警和工单状态会实时变化。WebSocket 可以让后端主动推送变化，前端不用一直轮询，提高实时性和效率。

### 11.8 为什么保留本地 vendor 文件？

答辩现场网络不一定稳定。如果依赖外部 CDN，页面可能无法加载。保留本地 Vue、ECharts、Leaflet 文件可以提高演示可靠性。

### 11.9 为什么根目录只保留 `start_system.bat`？

为了避免答辩时误启动半套系统。统一脚本同时启动后端和前端，检查端口和依赖，并自动打开页面。

### 11.10 为什么健康检查使用 SQLite？

健康检查主要验证代码和核心接口是否可用，不应依赖 MySQL 服务状态。SQLite 便于快速、稳定地执行本地自检。

## 13. 现场故障预案

| --- | --- | --- |

## 14. 代码讲解时的表达原则

答辩讲代码时不要逐行念代码，而要按“入口、职责、输入、处理、输出”讲。

推荐结构：

```text
这个文件负责什么
  -> 入口函数是什么
  -> 输入数据是什么
  -> 主要处理步骤是什么
  -> 输出给哪个模块
  -> 为什么这样设计
```

示例：

> `device_integration.py` 负责统一设备接入。它的输入是设备注册或读数上报请求，先校验设备编号和 API Key，再检查字段完整性，把时间和数值转换为标准格式，然后调用 `services.ingest_reading()` 写入读数并触发检测。这样设计的原因是 HTTP、MQTT、Webhook 和模拟器都能复用同一套接入逻辑。

# 答辩用系统总结与代码讲解

本文档用于毕业答辩时讲解整个系统的工程实现、运行流程和核心代码。内容基于当前项目源码和已有系统文档整理，只新增本文档，不改动任何原始源码或论文文档。

## 一、系统总体定位

本系统是一个面向物联网智能燃气表的异常检测与运维管理平台。系统的核心目标是把燃气表实时上报的多维时序数据转化为可解释、可处置、可追踪的异常告警和运维工单。

系统由五部分组成：

1. 设备接入：支持设备注册、HTTP 数据上报、MQTT 网关、运营商 Webhook 和虚拟设备模拟。
2. 数据处理：完成字段校验、类型转换、历史数据落库、滑动窗口构造和特征标准化。
3. 模型推理：使用 LSTM AutoEncoder 计算重构误差，并结合业务规则输出异常类型和严重程度。
4. 业务闭环：将异常检测结果保存为告警，并自动或手动形成工单，支持工程师接单和回单。
5. 前端展示：使用 Vue 3 单页应用展示仪表盘、设备、告警、工单、历史数据、模型版本和系统设置。

一句话讲系统：

> 设备数据进入后端后，系统先校验并落库，再按设备构造 24 点时序窗口，输入 LSTM AutoEncoder 计算异常得分；如果异常得分和业务规则共同判定异常，就生成告警和工单，并通过前端实时展示。

## 二、项目目录结构

项目根目录主要包含以下内容：

```text
src/                 Flask 后端、数据库模型、检测服务、训练模块、网关和业务逻辑
src/domain/          业务域门面，对外暴露更清晰的模块入口
frontend/            独立 Vue 3 前端应用
scripts/             健康检查和实验结果导出脚本

data/                训练数据、实验输出和运行数据目录
models/              LSTM 模型、标准化器、模型元数据和注册表
instance/            本地 SQLite 数据库实例目录
```

答辩时可以强调：系统采用前后端分离方式，后端提供 `/api/*` JSON 接口和 `/ws` WebSocket，前端独立运行并调用后端 API。

## 四、系统运行主流程

完整运行流程可以按下面讲。

1. 设备通过 `/api/device/register` 注册，系统生成或保存 API Key。
2. 设备通过 `/api/device/upload` 上报读数，请求头携带 `X-API-Key`。
3. 后端校验 API Key，确认设备合法。
4. 后端检查上报字段是否完整，并转换时间和数值类型。
5. 读数写入 `meter_readings` 表，同时写入物理数据记录表。
6. 系统更新设备在线状态和最后上报时间。
7. 系统提交异常检测任务。
8. 检测服务读取该设备最近读数。
9. 如果不足 24 条，使用启动阶段规则评分。
10. 如果达到 24 条，构造 24 点滑动窗口。
11. 使用 scaler 对 7 个特征标准化。
12. 调用 LSTM AutoEncoder 重构窗口。
13. 计算输入窗口和重构窗口的 MSE，作为异常得分。
14. 使用阈值和业务规则判断是否异常。
15. 检测结果写回 `MeterReading`。
16. 异常时创建或更新 `AnomalyEvent`。
17. 高价值异常自动创建或关联 `WorkOrder`。
18. WebSocket 推送最新仪表盘、设备、告警、工单和报表数据。
19. 前端页面实时更新。

## 五、核心配置代码讲解

配置集中在 `src/config.py`。

### 1. 路径配置

`BASE_DIR` 指向项目根目录，系统根据它创建数据、模型、文档和实例目录。

重要路径包括：

```text
DATA_DIR              data/
MODEL_DIR             models/
MODEL_REGISTRY_DIR    models/registry/
DOCS_DIR              docs/
INSTANCE_DIR          instance/
DATABASE_FILE         instance/gas_monitor.db
MODEL_FILE            models/lstm_autoencoder.pt
SCALER_FILE           models/scaler.pkl
META_FILE             models/train_meta.json
ACTIVE_MODEL_META_FILE models/active_model.json
```

这些路径把代码、数据、模型和文档分开管理。

### 2. 模型参数

核心模型参数：

```text
FEATURE_COLUMNS = [
    instant_flow,
    cumulative_usage,
    battery_voltage,
    signal_strength,
    valve_state,
    temperature,
    pressure
]
WINDOW_SIZE = 24
TRAIN_EPOCHS = 20
BATCH_SIZE = 32
LEARNING_RATE = 0.001
HIDDEN_SIZE = 32
NUM_LAYERS = 2
TRAIN_SPLIT = 0.8
```

答辩时重点解释 `FEATURE_COLUMNS` 和 `WINDOW_SIZE`。系统使用 7 个特征，每次取某台设备最近 24 条连续读数输入模型。

### 3. 数据库配置

`get_database_uri()` 支持两种方式：

- 如果设置 `DATABASE_URL`，直接使用该连接串。
- 如果 `DB_BACKEND=mysql`，使用 MySQL。
- 否则使用 SQLite 本地数据库。

这样既支持部署环境使用 MySQL，也支持本地测试快速切换 SQLite。

## 六、数据库模型代码讲解

数据库模型集中在 `src/database.py`，使用 Flask-SQLAlchemy 定义。

### 1. User

`User` 表用于系统登录和权限控制。

关键字段：

- `username`
- `username_encrypted`
- `username_lookup`
- `password_hash`
- `full_name_encrypted`
- `role`
- `engineer_id`
- `is_active`

系统采用“加密字段 + lookup 字段”的方式保存敏感信息。展示时使用解密字段，查询时使用 SHA-256 lookup，避免数据库中直接保存明文用户名、姓名、工号和电话。

### 2. Device

`Device` 表保存燃气表设备信息。

关键字段：

- `meter_id`：燃气表唯一编号。
- `name`：设备名称。
- `location`、`area`、`latitude`、`longitude`：位置与片区。
- `protocol`：接入协议，如 HTTP 或 MQTT。
- `api_key`：设备上报密钥。
- `is_enabled`：是否启用。
- `status`：在线、离线或告警等状态。
- `last_seen_at`：最后上报时间。

该表与读数、告警和工单都有关系，是系统业务链路的起点。

### 3. MeterReading

`MeterReading` 表保存每次设备上报的历史读数和检测结果。

原始读数字段：

- `timestamp`
- `instant_flow`
- `cumulative_usage`
- `battery_voltage`
- `signal_strength`
- `valve_state`
- `temperature`
- `pressure`

检测结果字段：

- `anomaly_score`
- `threshold`
- `predicted_label`
- `model_version`

答辩时可说明：读数表不仅保存原始数据，还保存模型推理结果，使每条异常判断都可以追溯到具体数据和模型版本。

### 4. PhysicalDataRecord

`PhysicalDataRecord` 保存物理设备上报记录，用于设备原始数据留存和数据保留策略。系统会根据配置清理超过保留天数的数据。

### 5. AnomalyEvent

`AnomalyEvent` 表保存异常告警。

关键字段：

- `device_id`
- `reading_id`
- `anomaly_type`
- `description`
- `score`
- `threshold`
- `severity`
- `status`
- `model_version`
- `confirmed_label`
- `handled_at`

该表把“模型结果”转成了“业务告警”。`confirmed_label` 用于人工反馈，例如确认异常、误报、忽略。

### 6. WorkOrder 和 WorkOrderRecord

`WorkOrder` 表保存工单主体，`WorkOrderRecord` 表保存工单流转记录。

工单状态包括：

- `pending`：待处理
- `assigned`：已派单
- `in_progress`：处理中
- `completed`：已完成

`WorkOrderRecord` 记录每次创建、派单、接单、回单的操作，便于答辩时说明系统具备完整流程追踪能力。

### 7. ModelMetadata

`ModelMetadata` 保存模型版本信息。

关键字段：

- `model_id`
- `model_path`
- `scaler_path`
- `meta_path`
- `feature_columns`
- `window_size`
- `hidden_size`
- `num_layers`
- `threshold`
- `accuracy`
- `precision_score`
- `recall_score`
- `f1_score`
- `is_active`

该表用于模型版本管理。系统不会只覆盖一个模型文件，而是记录每个候选模型的指标和激活状态。

### 8. TrainingRawDataRecord 和 TrainingCleanDataRecord

这两个表分别保存训练原始数据和清洗后数据，用于持续训练流水线。

## 八、设备接入代码讲解

设备接入主要在 `src/device_integration.py`。

### 1. 设备注册 register_device_endpoint()

该函数处理设备注册。

核心逻辑：

1. 从请求中读取 `meter_id`、名称、位置、协议、经纬度等信息。
2. 如果 `meter_id` 为空，抛出错误。
3. 如果设备不存在，创建新设备并生成 API Key。
4. 如果设备已存在，则更新设备基础信息。
5. 提交数据库事务。
6. 返回设备对象。

这保证了每台设备都有唯一编号和通信密钥。

### 2. 数据上报 accept_device_reading()

该函数处理设备上传读数。

核心逻辑：

1. 根据请求头中的 API Key 查询设备。
2. 如果设备不存在，返回无效密钥。
3. 调用 `_normalize_payload()` 校验和转换字段。
4. 调用 `ingest_reading()` 写入读数并触发检测。
5. 写入 `PhysicalDataRecord` 保存物理数据记录。
6. 清理过期物理数据。
7. 返回设备和读数。

### 3. 字段校验 _normalize_payload()

系统要求上报字段必须包含：

```text
timestamp
instant_flow
cumulative_usage
battery_voltage
signal_strength
valve_state
temperature
pressure
```

缺失字段会直接抛出错误。字段完整后，系统把时间转为 `datetime`，把流量、电压、信号、温度、压力等转成浮点数，把阀门状态转成整数。

答辩时可以强调：这是进入模型前的数据质量保障。

## 九、业务服务层代码讲解

核心业务逻辑集中在 `src/services.py`。

### 1. 设备状态刷新

`refresh_device_statuses()` 根据最后上报时间判断设备是否在线。

如果设备启用，并且 `last_seen_at` 在在线超时阈值内，则状态为 `online`；否则为 `offline`。

工程师在线状态由 `refresh_engineer_statuses()` 类似处理。

### 2. 仪表盘数据

`dashboard_snapshot()` 生成仪表盘需要的数据，包括设备汇总、告警、地图点位、趋势等。

前端不直接访问多个数据库表，而是调用该服务函数返回聚合后的 JSON。

### 3. 告警快照

`alerts_snapshot()` 查询异常告警并序列化为前端需要的结构，包括设备编号、异常类型、异常得分、阈值、严重程度、状态和时间。

### 4. 历史数据与重构分析

`device_history()` 返回设备历史读数。

`reconstruction_snapshot()` 调用检测服务的 `reconstruction_trace()`，返回原始序列、重构序列和误差序列，用于前端绘制重构分析图。

### 5. 工单管理

工单相关函数包括：

- `create_work_order()`
- `update_work_order()`
- `engineer_accept_work_order()`
- `engineer_complete_work_order()`
- `delete_work_order()`
- `work_orders_snapshot()`

`create_work_order()` 支持根据片区选择候选工程师。如果创建者是主管理员，会给更高派单优先级。工单创建后会写入 `WorkOrderRecord`，记录流程节点。

`engineer_accept_work_order()` 检查当前用户是否为对应工程师，并把状态改为 `in_progress`。

`engineer_complete_work_order()` 检查权限、填写处理结果、把状态改为 `completed`，同时更新关联告警的处理状态和人工确认标签。

### 6. 自动创建工单

`_auto_create_work_order_for_event()` 在出现异常告警时自动创建工单。

它会先检查该设备是否已有未完成工单。如果有，则关联现有工单；如果没有，则根据设备片区和工程师状态选择候选工程师，创建新工单。

这体现了系统的“异常告警到工单闭环”能力。

## 十、异常检测服务代码讲解

异常检测核心在 `src/detection_service.py`。

### 1. DetectionResult

`DetectionResult` 是一个数据类，封装检测输出：

```text
anomaly_score   异常得分
threshold       模型阈值
predicted_label 是否异常，0 正常，1 异常
anomaly_type    异常类型
description     异常描述
severity        严重程度
model_version   模型版本
```

### 2. DetectionService.ensure_ready()

该方法负责确保模型可用。

逻辑如下：

1. 确保模型目录存在。
2. 如果模型文件、标准化器或元数据不存在，则先运行训练流水线生成模型。
3. 获取当前 active model 版本。
4. 如果模型尚未加载或版本变化，则重新加载 scaler、meta 和 PyTorch 模型。
5. 设置模型为 `eval()` 推理模式。

答辩时可说明：系统支持模型热更新。当 active model 版本变化时，检测服务会重新加载模型。

### 3. score_recent_readings()

这是在线检测主函数。

如果读数不足 `WINDOW_SIZE=24`：

- 使用 `_bootstrap_score()` 进行规则评分。
- 设置临时阈值 0.85。
- 结合 `_has_operational_anomaly()` 判断是否异常。

如果读数达到 24 条：

1. 调用 `_frame_from_readings()` 转成 DataFrame。
2. 调用 `clean_dataset()` 清洗数据。
3. 调用 `transform_features()` 使用 scaler 标准化。
4. 取 7 个特征构造窗口。
5. 转成 PyTorch Tensor，形状为 `1 × 24 × 7`。
6. 输入 LSTM AutoEncoder 得到重构结果。
7. 计算 `torch.mean((reconstructed - tensor) ** 2)` 得到异常得分。
8. 读取模型元数据中的阈值。
9. 结合业务规则输出最终标签。
10. 推断异常类型、描述和严重程度。

这段代码正好对应论文中的模型推理流程。

### 4. reconstruction_trace()

该函数用于前端展示重构分析。

返回内容包括：

- 是否具备完整窗口
- 模型版本
- 阈值
- 特征列
- 时间戳
- 原始数据
- 重构数据
- 每个时间点误差
- 平均异常得分

前端历史数据页面可以用这些数据画出原始曲线、重构曲线和误差曲线。

### 5. 业务规则

`_has_operational_anomaly()` 判断是否存在明确业务异常，例如：

- 电池电压过低
- 信号强度过低
- 压力过低或过高
- 瞬时流量相对近期均值突增
- 瞬时流量相对近期均值突降

`_infer_anomaly_type()` 输出具体异常类型：

- 低电压告警
- 通信异常
- 流量尖峰
- 长时间流量静止
- 压力异常
- 综合异常
- 正常

答辩时可以说：模型负责给出异常分数，业务规则负责让异常结果更可解释。

## 十一、LSTM AutoEncoder 模型代码讲解

模型定义在 `src/model.py`。

核心类是 `LSTMAutoEncoder`。

结构包括：

1. `encoder`：LSTM 编码器，输入维度是 7，隐藏层维度默认 32，层数默认 2。
2. `decoder`：LSTM 解码器，输入是编码器输出的上下文向量。
3. `output_layer`：线性层，把隐藏状态映射回原始特征维度 7。

`forward()` 流程：

1. 输入形状为 `batch_size × sequence_length × feature_count`。
2. 编码器读取整个序列，输出最后一层隐藏状态。
3. 取最后一层 hidden，复制成与输入序列等长的上下文序列。
4. 解码器根据上下文序列生成重构隐藏序列。
5. 线性层输出重构后的 7 维特征。

可以用一句话解释：

> 编码器把 24 点多维时序数据压缩成一个上下文表示，解码器再根据这个表示还原原始序列，系统通过还原误差判断数据是否异常。

## 十二、在线检测调度代码讲解

在线检测调度在 `src/services.py` 的 `ingest_reading()` 和相关私有函数中。

### 1. ingest_reading()

该函数负责读数入库和触发检测。

流程：

1. 检查设备是否启用。
2. 创建 `MeterReading`。
3. 写入原始特征字段。
4. 更新设备 `last_seen_at` 和 `status`。
5. 提交数据库。
6. 根据配置选择同步检测或异步检测。
7. 返回读数对象。

### 2. 异步检测

系统使用 `ThreadPoolExecutor` 创建检测线程池。

如果 `ASYNC_DETECTION_ENABLED` 开启，系统会调用 `_submit_detection_task()`，把检测任务提交给后台线程。

这样设备上传接口可以更快返回，检测结果稍后写回数据库并通过 WebSocket 推送。

### 3. _run_detection_for_reading_once()

该函数执行单条读数对应的检测。

流程：

1. 根据 reading_id 读取读数。
2. 查询同设备、同时间之前的所有读数。
3. 调用 `detector.score_recent_readings()`。
4. 把异常得分、阈值、预测标签、模型版本写回读数。
5. 如果预测为异常，创建或更新 `AnomalyEvent`。
6. 调用 `_auto_create_work_order_for_event()` 自动生成或关联工单。
7. 提交数据库。
8. 推送实时更新。

这段是论文“实时推理、告警生成、工单闭环”的工程落点。

## 十三、训练模块代码讲解

训练逻辑在 `src/training_module.py`。

### 1. ContinuousTrainingService

这是持续训练后台服务。

它维护：

- 训练间隔
- 设备数量
- 每台设备记录数
- 随机种子
- 后台线程
- 停止事件
- 最近一次训练状态
- 最近一次训练结果或错误

`start()` 启动线程，`stop()` 停止线程，`status()` 返回前端展示所需状态。

### 2. run_training_pipeline()

这是训练流水线总入口。

流程：

1. 如果需要，生成训练原始数据。
2. 如果需要，清洗训练数据。
3. 调用 `train_model_from_clean_table()` 训练模型。
4. 返回每一步结果、数据量和训练数据健康状态。

### 3. train_model_from_clean_table()

这是模型训练主函数。

关键步骤：

1. 读取清洗后的训练数据。
2. 拟合 StandardScaler。
3. 标准化特征。
4. 构造滑动窗口。
5. 只选择正常窗口作为训练样本。
6. 按 `TRAIN_SPLIT` 划分训练集和验证集。
7. 使用 DataLoader 批量训练。
8. 创建 LSTM AutoEncoder。
9. 使用 Adam 优化器和 MSELoss。
10. 训练过程中记录 train loss 和 valid loss。
11. 使用 early stopping 保存最佳模型状态。
12. 使用验证集重构误差计算阈值。
13. 在全部窗口上评估 Accuracy、Precision、Recall 和 F1。
14. 保存模型、scaler 和元数据。
15. 注册候选模型版本。
16. 判断是否激活新模型。

### 4. 阈值计算

`_compute_threshold()` 使用验证集重构误差的 92 分位数作为阈值。

答辩时可以说：阈值来自模型在正常验证数据上的误差分布，不是人工拍脑袋设置。

### 5. 模型评估

`_evaluate_candidate()` 根据重构误差和阈值计算预测结果，再与窗口元数据中的真实异常标记比较，得到：

- TP
- TN
- FP
- FN
- accuracy
- precision
- recall
- f1

这些指标用于判断候选模型是否优于当前 active model。

## 十四、模型注册与版本管理

模型注册逻辑在 `src/model_registry.py`。

虽然答辩不一定需要逐行讲，但要说明它解决的问题：

2. 模型包包含 `.pt` 模型文件、`.pkl` 标准化器和 `.json` 元数据。

5. `activate_model_bundle()` 将更优模型切换为 active model。
6. `sync_active_model_metadata_to_db()` 把 active model 元数据同步到数据库。

## 十五、数据漂移监控

数据漂移相关代码在 `src/drift_monitor.py`。

它用于观察近期数据或异常得分是否相对历史分布发生明显变化。如果检测到漂移风险，可以触发持续训练服务。

答辩讲法：

> 燃气表数据分布会随季节、用户习惯和设备老化变化。漂移监控用于发现模型当前面对的数据是否与训练数据明显不同，一旦出现风险，就可以启动持续训练，避免模型长期不更新导致性能下降。

## 二十三、容易被问到的代码问题

### 1. 模型输入是什么？

模型输入是某台设备最近 24 条读数构成的窗口，每条读数包含 7 个特征，所以单个样本形状是 `24 × 7`。

### 2. 异常得分怎么计算？

模型输出重构后的 `24 × 7` 序列。系统计算重构序列和原始输入之间的均方误差 MSE，作为异常得分。

### 3. 设备上传后为什么返回 detection_status 可能是 pending？

因为系统支持异步检测。读数先入库，检测任务在后台线程执行。如果接口返回时检测尚未完成，就返回 `pending`；完成后前端通过刷新或 WebSocket 获取结果。

### 4. 为什么有 `model_version`？

因为系统支持模型版本管理。每条读数的检测结果都记录使用的模型版本，便于追溯问题、比较模型效果或回退。

### 5. 业务规则会不会替代模型？

不会。业务规则是对模型结果的复核和解释。模型负责发现复杂时序偏离，规则负责判断低电压、弱信号、压力异常、流量突变等明确业务异常。

### 6. 为什么要有 `TrainingRawDataRecord` 和 `TrainingCleanDataRecord` 两张表？

原始训练数据保留数据来源和原始状态，清洗数据用于模型训练。分开保存便于追踪数据处理过程，也便于检查清洗质量。

### 7. 如何保证前端实时更新？

后端在设备、告警、工单和报表变化后调用 WebSocket hub 广播事件。前端收到事件后更新状态并重绘图表。

### 8. 系统如何处理工程师在线状态？

工程师登录或前端上报在线状态时，系统更新 `last_online_at`。后台根据超时阈值判断工程师是否仍在线，并在派单候选排序时优先选择在线且负载低的工程师。

## 二十六、核心源码逐文件讲解

答辩时如果老师要求“讲代码”，建议按文件职责讲，不要从第一行读到最后一行。

### 1. `src/app.py`

该文件是 Flask 应用入口，负责创建应用、注册路由、初始化数据库、启动后台服务和提供 WebSocket。

重点讲解点：

- `create_app()`：创建 Flask 应用，加载配置，初始化数据库和服务。
- `register_routes()`：集中注册页面路由和 `/api/*` 接口。
- `login_required()` 和 `roles_required()`：实现登录校验和角色权限控制。
- `/api/device/register`、`/api/device/upload`：设备注册和数据上报入口。
- `/ws`：前端实时推送通道。

答辩表述：

> `app.py` 相当于系统入口和控制层，前端页面、设备接口、管理接口和 WebSocket 都从这里进入，然后再调用服务层完成具体业务。

### 2. `src/database.py`

该文件定义数据库实体，支撑论文中的 E-R 图。

主要模型包括 `User`、`Device`、`MeterReading`、`PhysicalDataRecord`、`AnomalyEvent`、`Engineer`、`WorkOrder`、`WorkOrderRecord`、`TrainingRawDataRecord`、`TrainingCleanDataRecord` 和 `ModelMetadata`。

讲解重点：

> 数据库设计保证了读数能追溯到设备，告警能追溯到读数和模型版本，工单能追溯到告警和工程师，人工反馈能进入训练数据闭环。

### 3. `src/services.py`

该文件是业务服务层，负责把数据库、检测服务、告警、工单、报表和实时推送串起来。

重点函数：

- `dashboard_snapshot()`：生成仪表盘汇总数据。
- `devices_snapshot()`：生成设备列表数据。
- `alerts_snapshot()`：生成告警列表。
- `work_orders_snapshot()`：生成工单列表。
- `create_work_order()`、`update_work_order()`：工单创建和更新。
- `engineer_accept_work_order()`、`engineer_complete_work_order()`：工程师接单和完成工单。
- `ingest_reading()`：读数入库和检测任务提交的核心入口。
- `_run_detection_for_reading_once()`：执行单条读数对应的检测和告警生成。
- `_auto_create_work_order_for_event()`：异常事件触发工单。

### 4. `src/detection_service.py`

该文件实现在线异常检测。核心类 `DetectionService` 负责加载 active model、标准化器和阈值；`DetectionResult` 封装异常得分、阈值、标签、异常类型、严重程度和模型版本。

检测流程为：读取设备最近读数，不足 24 点时使用启动阶段规则评分；达到 24 点后构造特征矩阵，使用 scaler 标准化，调用 LSTM AutoEncoder 重构，计算 MSE，与阈值比较，并叠加业务规则输出异常类型。

### 5. `src/model.py`

该文件定义 LSTM AutoEncoder 模型结构。编码器 LSTM 把输入时间序列压缩为隐状态，解码器 LSTM 根据隐状态重构序列。模型训练目标是让正常数据重构误差尽可能小，异常数据因偏离正常模式，重构误差会升高。

### 6. `src/model_registry.py`

该文件负责模型版本生命周期。`stage_model_bundle()` 保存候选模型文件、标准化器和元数据；`should_activate()` 比较候选模型和当前模型指标；`activate_model_bundle()` 切换 active model；`active_model_metadata()` 读取当前模型元数据。

答辩表述：

> 模型注册表避免直接覆盖线上模型。新模型先训练评估，再根据指标决定是否切换，便于追踪和回退。

### 7. `src/training_module.py`

该文件负责训练和持续训练。`ContinuousTrainingService` 是后台持续训练服务；`generate_training_data()` 生成训练样本；`clean_training_data()` 清洗训练数据；`train_model_from_clean_table()` 从清洗表训练模型；`run_training_pipeline()` 串联完整训练流水线；`_evaluate_candidate()` 计算候选模型 Accuracy、Precision、Recall、F1 等指标。

### 8. `src/device_integration.py`

该文件统一设备接入流程。`register_device_endpoint()` 处理设备注册，`accept_device_reading()` 统一接收设备读数，`_normalize_payload()` 做字段标准化，`_coerce_datetime()` 做时间解析。该模块让 HTTP、MQTT、运营商 Webhook 和模拟设备最终进入同一条处理链路。

### 9. 前端文件

`frontend/src/main.js` 是 Vue 单页应用主文件，负责页面状态、API 请求、WebSocket、ECharts、Leaflet、表单和交互逻辑。`frontend/src/styles.css` 定义系统界面样式，保证仪表盘、表格、卡片、图表和移动端布局一致。

## 三十一、代码答辩高频追问

### 1. 为什么要有服务层？

因为路由层只适合处理请求和响应，真正的业务逻辑包括数据入库、检测、告警、工单和推送。如果全部写在路由里，代码会难以维护。服务层把业务逻辑集中起来，使 HTTP、MQTT、模拟器等入口可以复用同一套流程。

### 2. 为什么要异步检测？

设备上报接口需要快速返回，模型推理和告警生成可能耗时。异步检测可以先保证数据入库，再由后台线程完成推理和事件生成，避免上报接口阻塞。

### 3. 数据不足 24 条时怎么办？

LSTM AutoEncoder 需要完整窗口，数据不足时系统采用启动阶段规则评分。等设备累计读数达到窗口长度后，再进入完整模型推理。

### 4. 为什么要保存物理数据记录？

`MeterReading` 关注模型检测和业务展示，`PhysicalDataRecord` 更偏原始接入归档。分开保存有利于追踪原始上报和后续处理结果。

### 5. 为什么要有人工反馈？

异常检测系统需要持续改进。人工反馈可以把现场确认结果沉淀为标注样本，后续用于模型评估、规则优化和持续训练。

### 6. 前端为什么使用 WebSocket？

设备读数、告警和工单状态会实时变化。WebSocket 可以让前端及时刷新，不必完全依赖定时轮询，提高实时性。

### 7. 系统安全体现在哪里？

设备上报使用 API Key，运营商 Webhook 可配置令牌；用户登录使用密码哈希；身份敏感字段使用字段级加密和 lookup 查询；路由层通过登录和角色装饰器控制权限。

## 三十二、接口设计答辩讲法

如果老师问“前后端和设备之间怎么通信”，可以按三类接口说明。

第一类是浏览器用户接口。前端通过 `/api/auth/login` 登录，后端写入 Session；后续管理接口依赖 Session 和角色权限判断。前端每次请求都带上登录态，后端再通过 `login_required` 和 `roles_required` 做二次校验。

第二类是设备接入接口。设备先调用 `/api/device/register` 注册，系统保存设备编号并生成通信密钥；设备上报时调用 `/api/device/upload`，请求头必须携带 `X-API-Key`。后端根据 `meter_id` 和密钥确认设备身份，校验通过后才允许数据入库和触发检测。

第三类是平台和实时接口。运营商平台通过 `/api/carrier/webhook/<provider>` 转发数据，生产环境可以配置 `CARRIER_WEBHOOK_TOKEN`；前端通过 `/ws` 接收设备状态、告警、工单和设置状态变化，减少页面轮询。

接口设计可以这样总结：

> 系统接口分为用户 Session、设备 API Key 和运营商 Token 三套入口。这样既能支持浏览器用户管理，也能支持物联网设备自动上报，还能扩展第三方平台接入。所有入口最终都会进入统一服务层，避免不同协议各自实现一套检测逻辑。

答辩时可举的关键接口：

| 场景 | 接口 | 说明 |
| --- | --- | --- |
| 登录 | `/api/auth/login` | 管理员和工程师登录 |
| 会话检查 | `/api/auth/session` | 前端刷新后恢复登录状态 |
| 设备注册 | `/api/device/register` | 创建燃气表并生成密钥 |
| 设备上报 | `/api/device/upload` | 写入读数并触发检测 |
| 仪表盘 | `/api/dashboard` | 返回设备、告警、工单汇总 |
| 历史曲线 | `/api/history` | 返回读数和检测结果 |
| 重构分析 | `/api/reconstruction` | 返回原始序列、重构序列和误差 |
| 告警反馈 | `/api/alerts/<id>/feedback` | 人工确认异常或误报 |
| 工单处理 | `/api/work-orders/<id>/accept`, `/api/work-orders/<id>/complete` | 工程师接单和回单 |
| 系统设置 | `/api/training`, `/api/simulator`, `/api/mqtt`, `/api/drift` | 控制训练、模拟、MQTT 和漂移监控 |

## 三十三、开发维护答辩讲法

如果老师问“后续怎么维护和扩展”，可以从分层职责回答。

系统没有把所有逻辑写在一个文件里，而是按职责拆分：

- `app.py` 负责入口、路由、权限和 WebSocket。
- `services.py` 负责核心业务流程，例如设备、告警、工单、报表和推送。
- `device_integration.py` 负责统一设备接入。
- `detection_service.py` 负责模型推理。
- `training_module.py` 和 `model_registry.py` 负责训练和模型版本。
- `database.py` 负责表结构。
- `frontend/src/main.js` 负责页面状态、接口调用和图表。

新增功能时的维护原则：

1. 新接口先放到 `app.py` 的路由层，但路由层只接收参数和返回结果。
2. 业务判断写入 `services.py` 或对应模块，便于 HTTP、MQTT、模拟器复用。
3. 新数据字段先修改数据库模型，再同步序列化、前端展示、接口文档和测试文档。
4. 新设备协议先做字段映射，再进入统一设备上报流程。
5. 新模型不能直接覆盖旧模型，应先注册候选版本，再决定是否切换 active model。

可以用一句话回答：

> 后续维护的关键是保持入口层、业务层、算法层、数据层和前端展示层的边界。这样新增协议、新增页面或更换模型时，不会牵一发而动全身。

## 三十五、部署运维答辩讲法

如果老师问“系统如何部署”，可以按下面顺序回答。

部署环境建议使用 Windows 10/11 或 Windows Server，Python 3.10 以上，数据库使用 MySQL 8.x。开发和演示可以切换到 SQLite，正式部署建议使用 MySQL。

部署步骤：

1. 安装依赖：`pip install -r requirements.txt`。
2. 创建 MySQL 数据库 `gas_monitor`。
3. 配置环境变量，包括数据库、`APP_SECRET_KEY`、`APP_CREDENTIAL_SECRET`、前端地址和 MQTT 参数。
4. 启动后端：`.\start_system.bat`。
5. 启动前端：`.\start_frontend.bat`。
6. 登录系统，检查仪表盘、设备、告警、工单、历史数据和设置页。

部署时最重要的风险点：

- `APP_CREDENTIAL_SECRET` 不能随意更换，否则历史加密字段无法解密。
- MySQL 密码不应写入公开仓库，生产环境应通过环境变量注入。
- `models/` 和 `models/registry/` 要和数据库一起备份，否则历史检测结果难以复现。
- 前端地址变化时，需要同步修改 `FRONTEND_URL` 和 `FRONTEND_ORIGINS`。
- MQTT 接入前要确认 Broker 地址、端口、用户名、密码和主题都正确。

可以这样总结：

> 系统部署不只是启动 Web 服务，还要保证数据库、密钥、模型文件、前端来源和后台任务配置一致。尤其是模型版本和加密密钥，需要作为运维资产长期保管。

## 三十六、现场故障预案

答辩现场如果系统出现问题，可以按现象快速定位。

| 现象 | 优先检查 | 现场说法 |
| --- | --- | --- |
| 前端打不开 | `start_frontend.bat` 是否启动，端口是否为 5173 | 前端是独立静态服务，需要单独启动 |
| 后端接口无响应 | `start_system.bat` 是否启动，5000 端口是否占用 | 后端提供 `/api/*` 和 `/ws` |
| 登录失败 | 默认账号、Session、数据库连接 | 登录依赖数据库用户表和 Session |
| 页面无数据 | 是否开启虚拟设备，数据库是否有读数 | 可通过模拟器生成演示数据 |
| 没有异常得分 | 单台设备是否达到 24 条读数，模型和 scaler 是否存在 | LSTM 推理需要完整窗口 |
| 设备上报失败 | `X-API-Key`、`meter_id`、字段完整性 | 设备接口有密钥和字段校验 |
| WebSocket 不刷新 | 浏览器控制台、`/ws` 连接、后端日志 | 实时刷新是增强能力，接口刷新仍可兜底 |
| 模型文件缺失 | `models/`、模型注册表、持续训练 | 系统可先用规则逻辑运行，再训练候选模型 |

现场演示如果时间紧，可以优先展示静态已有数据和页面，再说明完整链路由设备注册、上报、检测、告警和工单组成。
